Git Repository Lifecycle

Git follows a structured workflow from code creation to version control. Developers work in the Working Area, move changes to the Staging Area, commit them to the Local Repository, and finally synchronize with GitHub.

Phase 1: Local Repository Initialization

  1. Create a project directory.
  2. Initialize Git repository.
  3. Configure developer identity.
mkdir project-name

cd project-name

git init

git config --global user.name "Your Name"

git config --global user.email "your@email.com"

Phase 2: Local Workflow

Changes are first created in the Working Area, then moved to the Staging Area before being committed. 

git status

git add app.py

git add .

git commit -m "Initial Commit"

git log

Workflow Explanation

Area Purpose
Working Area Where code is created or modified
Staging Area Temporary area before commit
Commit Area Stores version history locally

Key Takeaways

  • Git tracks code history through commits.
  • Working Area → Staging Area → Commit Area is the core workflow.
  • GitHub stores repositories remotely.
  • README.md documents projects.
  • .gitignore prevents unnecessary files from being tracked.
  • PAT and SSH are secure authentication methods.
  • SSH provides passwordless Git operations.

Practice: Step-by-Step Action Plan

Phase 1: Local Repository Initialization

  1. Create a Project Directory: mkdir project-name
  2. Initialize the Repository: git init
  3. Configure Developer Information:
    • git config --global user.name "Your Name"
    • git config --global user.email "yourname@example.com"

Phase 2: Local Workflow (Working Area to Commit Area)

  1. Create and Modify Files: Work in the Working Area.
  2. Check Repository Status: git status
  3. Move Files to Staging:
    • Single file: git add <filename>
    • All files: git add .
  4. Commit Changes: git commit -m "Your descriptive message"
  5. Review History: git log

Phase 3: Working with Remote Repositories (GitHub)

  1. Create a GitHub Repository: Public vs Private, with optional README.md and .gitignore.
  2. Clone a Repository: git clone <repository-url>

Phase 4: Security and Authentication Setup

Git requires secure authentication methods for private repositories:

  • Method A: Personal Access Token (PAT)
    1. Generate token in GitHub Settings.
    2. Set expiration and permissions.
    3. Copy and use token in place of password.
  • Method B: SSH Key
    1. Generate keys: ssh-keygen -t rsa -C "your_email@example.com"
    2. Copy public key (id_rsa.pub).
    3. Add to GitHub Settings → SSH keys.
    4. Clone via SSH URL (git@github.com...).
