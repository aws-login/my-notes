Git Stash, Tag & Blame

Git Stash temporarily saves unfinished work, Git Tags are used for versioning and release management, and Git Blame helps identify who modified specific lines of code.

Git Stash

Key Takeaways

  • Git Stash temporarily saves unfinished work.
  • Stash Apply restores work while keeping stash history.
  • Stash Pop restores work and removes the stash.
  • Tags are commonly used for release management.
  • Annotated tags contain additional metadata.
  • Tags are not pushed automatically with normal git push.
  • Git Blame helps identify code ownership.
  • GitHub provides a browser-based blame interface.

Module 1: Managing Temporary Work with Git Stash

When you are in the middle of a task and must switch to an emergency bug fix without committing incomplete code, use Git Stash to save your working area changes temporarily.

  1. Identify Uncommitted Changes: Check your current status to see modified files in the working directory.
  2. Save to Temporary Storage: Move your working area changes into the stash.
    Command: git stash
  3. Verify Stash History: View the list of all stashed changes and their unique indices.
    Command: git stash list
  4. Restore Stashed Work: Once the emergency task is finished, bring your code back.
    • Option A (Keep in Stash): Use apply to bring changes back while keeping the record in the stash.
      Command: git stash apply <stash_id>
    • Option B (Remove from Stash): Use pop to bring changes back and immediately delete them from the stash.
      Command: git stash pop <stash_id>
  5. Clean Up: Manually delete specific stashes or clear the entire storage.
    Command (Single): git stash drop <stash_id>
    Command (All): git stash clear

Module 2: Versioning and Labeling with Git Tag

Tags act as labels for specific commits, primarily used to mark release versions (e.g., v1.0, v2.0).

  1. Determine Tag Type:
    • Light-weight Tag: A simple pointer to a commit.
    • Annotated Tag: A detailed tag containing a message, author, and timestamp.
  2. Apply Tag to a Commit:
    • Light-weight Syntax: git tag <version_name> <commit_id>
    • Annotated Syntax: git tag -a <version_name> -m "<message>" <commit_id>
  3. Verify Tags: List all existing tags or check a specific tag's details.
    Command: git tag
  4. Sync with Remote Repository: By default, tags are not pushed with standard push commands.
    • Push Specific Tag: git push origin <tag_id>
    • Push All Tags: git push origin --tags
  5. Remove Tags: Delete labels if they are no longer needed.
    Command: git tag -d <tag_id>

Module 3: Auditing Changes with Git Blame

Use this tool for debugging, tracking, and auditing to see which developer modified specific lines of a file.

  1. Investigate File History: Display the author, commit ID, and timestamp for every line in a file.
    Command: git blame <file_name>
  2. Narrow Down the Search: If a file is large, inspect only specific line ranges.
    Command: git blame -L <start_line>,<end_line> <file_name>
  3. Analyze on Web Interface: For a visual "color-coded" view from oldest to newest changes, use the Blame button on the GitHub file UI.
